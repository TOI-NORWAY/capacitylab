# Feedforward Model Training Guide

This document explains how the published feedforward model (FF-GlobalCapacity) is trained in its original internal repository. This document is meant to give background information on the model that is available in the GUI.

The training entry point is:

- src/train_flow_speed_v3_globalcap.py

## 1. What This Training Produces

The script trains two models in one run:

- FF-GlobalCapacity (feedforward network)
- GNN-GlobalCapacity

For the GUI publication workflow, only the FF artifact is used:

- models/best_ff_flowspeed_v3_globalcap.pth

## 2. Required Input Data

Training expects two inputs:

1. MATSim network XML
- Default path: model/output_network.xml
- Used to build link topology and static link features.

2. MATSim simulation CSV files
- Searched recursively under dataset root.
- Default dataset root: ny_data/GWE
- Expected filename: output_dataframe_30_hours_kdv_speed.csv

### Required CSV columns

- run_id (generated by script from folder path)
- network_variant
- hour
- link_id
- capacity
- flow_exit_link (default flow target)
- speed (default speed target)

Notes:
- The script filters to daytime hours 5 through 22.
- link_id values must also exist in model/output_network.xml to be used.

## 3. Reproducible Command

Run from repository root:

```bash
python src/train_flow_speed_v3_globalcap.py --model-name flowspeed_v3_globalcap
```

Useful reproducibility options:

```bash
python src/train_flow_speed_v3_globalcap.py \
  --model-name flowspeed_v3_globalcap \
  --dataset-root ny_data/GWE \
  --network-file model/output_network.xml \
  --ff-epochs 120 \
  --ff-batch-size 2048 \
  --run-id my_repro_run
```

Important:
- This script does not currently have an FF-only mode; it trains FF and GNN sequentially.
- If you only need FF quickly, reduce GNN workload (for example, lower gnn epochs).

## 4. Train/Validation/Test Split Logic

The split is scenario-level (not row-level) to reduce leakage.

Scenario key:

- (run_id, network_variant)

Split proportions:

- train: 70%
- validation: 10%
- test: 20%

Randomness:

- Seed: 42 (used for scenario shuffle and row cap sampling).

Context key for global-capacity modeling:

- (run_id, network_variant, hour)

Each context gets a full network capacity vector (length = number of links).

## 5. Feature Engineering for FF

### 5.1 Static link features (10 per link)

From model/output_network.xml:

- length
- freespeed
- permlanes
- euclidean length
- normalized direction dx
- normalized direction dy
- midpoint x
- midpoint y
- sin(angle)
- cos(angle)

### 5.2 Hour features

- One-hot encoding for daytime hours 5..22
- Dimension: 18

### 5.3 Capacity context features

For each sample/link:

- local capacity (scaled)
- upstream average capacity (scaled)
- downstream average capacity (scaled)
- local minus upstream average
- local minus downstream average

Plus full global capacity vector for the sample context:

- one scaled capacity value per link in the network

### 5.4 FF input vector

The FF model input is:

- hour one-hot (18)
- static features (10)
- local/neighborhood capacity features (5)
- global capacity vector (n_links)

So input dimension is:

- 18 + 10 + 5 + n_links

## 6. Scaling and Targets

Scalers (fit on train split only):

- StandardScaler for static features
- StandardScaler for capacity
- StandardScaler for flow target
- StandardScaler for speed target

Targets are 2D:

- y[:, 0] = scaled flow
- y[:, 1] = scaled speed

## 7. FF Neural Network Setup

Model class:

- FFGlobalCapacity

Architecture:

- Linear(input_dim, 512)
- ReLU
- Dropout(0.2)
- Linear(512, 256)
- ReLU
- Dropout(0.2)
- Linear(256, 2)

Default training hyperparameters:

- ff epochs: 120
- ff batch size: 2048
- learning rate: 1e-3
- weight decay: 1e-5
- optimizer: Adam
- loss: MSELoss on 2 outputs (equivalent equal weighting of flow/speed)
- gradient clipping: max norm 1.0

Runtime options:

- AMP enabled by default on CUDA
- torch.compile enabled by default on CUDA

## 8. Model Selection During Training

- Validation loss is computed each epoch.
- Best checkpoint is selected by minimum validation loss.
- The best state is used for final test prediction and artifact export.

## 9. Output Artifacts

For each run, outputs are written under:

- outputs/<model-name>/runs/<run-id>/

Key files for FF reproducibility:

- models/best_ff_flowspeed_v3_globalcap.pth
- data/artifacts_flowspeed_v3_globalcap.pkl
- data/scalers_flowspeed_v3_globalcap.pkl
- data/model_configs_flowspeed_v3_globalcap.pkl
- data/history_flowspeed_v3_globalcap.csv
- data/test_predictions_flowspeed_v3_globalcap.csv
- data/metrics_flowspeed_v3_globalcap.csv
- config/run_config.json
- run_manifest.json

Also maintained at model root:

- latest_run.txt
- run_index.csv

## 10. Practical Reproduction Checklist

1. Use same script and same repository revision.
2. Keep seed at 42.
3. Use same dataset root and network XML.
4. Keep daytime-hour filter (5..22).
5. Keep target columns flow_exit_link and speed.
6. Keep FF hyperparameters and optimizer settings.
7. Use the best checkpoint chosen by validation loss.
8. Export and retain all four GUI-required files:
- best_ff_flowspeed_v3_globalcap.pth
- artifacts_flowspeed_v3_globalcap.pkl
- scalers_flowspeed_v3_globalcap.pkl
- model_configs_flowspeed_v3_globalcap.pkl
